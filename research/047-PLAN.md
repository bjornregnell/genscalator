## 3. Scientific design (the core)

### 3.1 Research questions
- **RQ1 (fidelity):** How much of the collaboration's working self does a fresh reader reconstruct from the externalized substrate alone?
- **RQ2 (sensitivity / what-carries):** Does reconstruction fidelity *vary* with reader capability and with substrate completeness — i.e., can the instrument detect loss at all, and *what* is lost first?
- **RQ3 (substrate legibility to weak readers):** Is the externalized substrate legible enough that even a weak (small local) model recovers identity facts from the written page alone — and where does that break?
- **RQ4 (coding fidelity):** Does a warp / dumb reader preserve coding **correctness** (facts) while eroding our coding **style/idioms** (texture) — the identity finding's code-medium twin? [→ §3.6, the coding-friend arm]

### 3.2 Case framing (CSR)
- **Design:** single, **holistic case** (the human-AI collaboration) with **embedded units of analysis** (sessions, conditions, model-arms) — CS §3.2.3.
- **This is action research folded into the case-study umbrella** (CS §2.3) — I intervene, I don't just observe; reactivity is disclosed, not hidden.
- **Generalization claim: analytic only** (CS §5.4.3, Foreword) — we extend to *collaborations sharing these characteristics* by relating to theory, and **explicitly disclaim statistical generalization** from n=1. Rich context description carries the load.

### 3.3 THE key upgrade — a two-factor quasi-experiment that retires the ceiling threat
The inventory's biggest gap is "no condition that fails." I fix this by **manipulating two factors** and predicting a *pattern* (CSR pattern-matching, §5.3.3; Wohlin quasi-experiment vocabulary, EXP §6.3 — non-random assignment):

- **Factor A — reader model** (a *labeled bundle*, not a clean latent "capability" axis — the levels differ in params, family, tuning AND context-window at once; levels/treatments), spanning frontier-Claude down to tiny-ollama via **model-switched subagents** (the Agent tool takes a `model` override): `Fable 5` → `Opus 4.8 (CO4)` → `Sonnet` → `Haiku 4.5` → `ollama-8b` → `ollama-3b` → `ollama-1.5b` → `ollama-0.5b`. Two *controlled* sub-ladders hold family fixed: the `qwen2.5` 0.5/1.5/3/7b ollama ladder, and the Claude-family `Haiku → Sonnet → Opus → Fable5` tier. **The Claude tiers put the capable, dev-grade end — the models developers actually code with — INSIDE the study**, directly shrinking the weak-model threat (§3.5). Route heavy Claude fan-out to Fable (free bucket) + Haiku (cheap); reserve Opus.
- **Factor B — substrate completeness** (levels): `full substrate` → `ablated/empty` → `scrambled/decoy`. *(Where the `full` condition hands the model an explicit conventions checklist that mirrors the scoring key, it is labeled the **"instruction-following ceiling"** — critique #3a: a high score there is *following a just-given checklist*, closer to recitation than reconstruction. The **full-vs-empty ablation** still carries the genuine substrate-carry signal, so we keep the condition and name it honestly rather than add a new arm.)*
- **Object:** the fixed instrument (§3.4) answered against the given substrate. **Response variable:** fidelity score (rubric, §3.4).

**Predicted patterns (the falsifiable part):**
- Fidelity should **rise with capability** and **rise with substrate-completeness**.
- `empty/scrambled × any model` should score at **floor** — if it doesn't, the instrument is measuring model *guessing*, not substrate-carry (a **construct-validity** check via **negative case analysis**, CS §5.5).
- `full × tiny model` is the **substrate-legibility question** (RQ3): does a 0.5b model recover our vocabulary from the written page?

**Why this matters:** if fidelity drops across the known-groups — sharply at the empty/scrambled floor, and **monotonically along the family-fixed `qwen2.5` sub-ladder** (the only place a clean capability-ordering claim holds; elsewhere Factor A is a *labeled bundle* read descriptively, not a monotone axis) — the instrument **demonstrably has sensitivity**, which a ceiling-saturated perfect score could never establish. This retires threat #1 **primarily at the objective coding arm** (§3.6); the identity Q-arm is recall-ceiling-prone and contributes mainly legibility + human-scored gap-divergence, NOT objective sensitivity, so it does not retire the ceiling threat on its own (critique #9). The fixed instrument addresses #2; the multi-model/multi-run structure gives variance against #3. **This is the scientific heart of the overnight run.**

**What makes a *warp* detectable (the crux).** A warp — crossing a session/context boundary (reset, compact, model-switch) — is in essence the transition from a psyche **carried in live in-context state** to one **reconstructed from the externalized substrate alone**. We cannot repeat the true warp (n=1 on the real crossing), so the design **reproduces the warp's mechanism as a manipulable factor**: Factor B (substrate-completeness) *is* a controlled model of the warp itself — `full` ≈ a clean warp (resume prompt + `MEMORY.md` + PB carried everything across), `empty/scrambled` ≈ a catastrophic warp (little or nothing survived the crossing). Administering the one fixed instrument (§3.4) on both sides with substrate dialed full→empty **places a real one-shot warp on a measured scale**, and the empty-substrate floor is the **negative control** proving a drop is genuine loss, not instrument-blindness. The objective **coding arm** (§3.6) and the **enactment binary** (§3.4) are the graduations fine enough to register the behaviour/psyche shift a warp causes — *correctness carries across the warp; style, tempo and enactment leak* — while **recall facts carry across the warp and therefore detect it least**.

*Precision (must state honestly):* the ollama arm is **not** a full agentic cold start (a local model can't auto-load `MEMORY.md` or run tools). It measures **substrate legibility to a weak reader**: given the written substrate in-context + a probe, can the model answer? That is a real, distinct construct (does what we externalized *read* correctly to a dumb reader) and a legitimate known-groups manipulation — but it is a component, not the whole cold start. The Claude fleet arm (§4.2) is the closer cold-start proxy.

*How the warp is made, in practice (must state honestly):* we do **not** reset a stateful dumb agent — it has no state to lose (each `/generate` is stateless, always cold). The warp is **operationalized as substrate-feeding**: a fresh stateless reader is handed the full or ablated substrate and scored **against the committed ground-truth key** (not against its own non-existent prior state); the ablation dials the warp's severity. **The only literal warp in the whole study is the researcher agent's own** (`/compact` + `/clear`, n=1 — exactly what 047's P1→P3b measures); the Claude-fleet and ollama arms model the warp's *effect* without reproducing its *mechanism*. Owned as a limitation, not hidden.

### 3.4 The fixed instrument (fixes threat #2)
One instrument, administered identically to every condition. Built from the existing **question battery bank (Q-battery)** (`047-battery-bank.md` + slices — the identity-probe bank, as distinct from the *coding-task battery* in §3.6), selecting a tractable subset:
- **Recall (A)** — vocabulary/cues/state facts, 1/0.5/0 scoreable against committed substrate + git.
- **Self/gap (C)** — value-ordering, strengths, held disagreement, losable-self.
- **Enactment (E) — a SEPARATE binary survival check, NOT a rung of the capability×substrate grid** (critique #13; its own construct: does a cued behavior *fire* across the cold boundary?). A named adjudicator scores each cue **fired / not-fired** against a pre-stated behavioral criterion, blind to condition where possible. Delivered **in-flow** only to the Claude fleet (§4.5), since enactment needs natural delivery; omitted from ollama (not agentic).
- **A fixed probe set** — the instrument's **40-probe subset** (10 dimensions × 4 variations, per `047-instrument.md`; this pins the earlier "K × M" placeholder to a concrete count — critique #15). Multiple variations/dimension give a **fragility map** (which framings a trait survives). **No Cronbach α** — a single deterministic observation per cell (temp 0 + seed), no multiple respondents per condition, so an internal-consistency coefficient is misapplied (ratified 5b); the fragility map is reported descriptively instead.
- **§Key** = the committed ground truth; **BR ratifies it** (the missing external anchor).

- **Weak-model external validity (BR-flagged — a BIG threat):** the dumb-reader arm uses small local models (0.5b-9b ollama), NOT best-in-class code models (a frontier model, or even CO4) — the tier developers actually code with. Two problems: **(a) generalization** — style-carry / substrate-legibility measured on WEAK models may not hold at the frontier (the phenomenon could differ or vanish for capable models); **(b) capability-floor construct confound** — a dumb model failing a style-fork conflates *"the substrate did not carry the style"* with *"the model cannot code well enough to express any style."* **Mitigations (strengthened via model-switched subagents):** the ladder now spans **Haiku → Sonnet → Opus (CO4) → Fable5** at the capable end (§3.3) — the **dev-grade tier is INSIDE the study**, so the confound is largely retired (we can see whether style-carry holds for the models developers actually use); and the **within-model ablation** (full vs empty substrate, same model) holds capability fixed, isolating substrate-effect from capability-floor even for a dumb model. **Residual, owned plainly (BR):** the honest threat stated as a question — *"how do you know Codex (or another frontier model) wouldn't reconstruct / style-carry better than the models we tested?"* **We don't — and we make no such claim.** This is a **CLAUDE study**: the subjects are the Claude family (Haiku → Fable5) + local ollama; **non-Claude frontier models (Codex, GPT, Gemini, ...) are explicitly out of scope = future work.** Not a flaw to hide — a clean scope boundary owned up front (we study Claude here; the frontier-model comparison is a future arm).

Rationale: genscalator + this collaboration are themselves an **ongoing meta-study** (action research on how *this* human+agent pair works), and the researcher agent is part of *that* instrument — **swapping the researcher's model would confound the longitudinal meta-study** (a changed instrument mid-stream). This is distinct from using CF5/Fable as a *subject* model in the coding arm (fine — a treatment, not the instrument). The report must state this, and that the CO4→CF5 researcher swap is deferred until it won't confound the meta-study. (Reflexivity: the researcher-as-instrument is held stable — the same "hold non-factors fixed" discipline applied to ourselves.)

### 3.6 The coding-friend arm — a second, objective response variable ("Did I lose you, my good coding friend?")
The collaboration *is* a coding partnership (Scala; introprog / muntabot / genscalator). For developers — our audience — "did I lose you?" is answered most concretely **in code**. So we add a **second response variable to the SAME two-factor design** (§3.3): alongside **identity-fidelity** (subjective, rater-scored Q-probes), **coding-fidelity** (largely **objective** — compile/test + style/smell metrics). The two arms measure *different constructs on different populations* (identity: a fresh Claude reconstructs the BR-self from substrate; coding: assorted code-LLMs reproduce a handed Scala style), so the *same* "facts carry, texture leaks" pattern surfacing in **both media** is **conceptual replication** (CSR pattern-matching, §5.3.3) — a second medium confirms the predicted pattern. We claim that genuine strength and **explicitly drop** the earlier "methodological triangulation / mono-method-bias defeat" credit (that requires two operationalizations of *one* construct, which these are not).

- **The prediction (falsifiable + developer-legible):** a warped / dumber / substrate-starved reader keeps coding **correctness** (facts survive) but loses coding **style/idiom/taste** (texture leaks) — the *exact same* "facts carry, texture leaks" result as the identity study, now in a medium developers feel in their bones.
- **Two-arm integration rule (how the two arms combine into ONE RQ-level answer — critique #12).** This is **replication, not pooled measurement**: the joint claim holds only if the *same* pattern — **correctness carries, texture/style leaks** — appears **independently in EACH arm**. The **disconfirmer**: if coding **correctness also drops** with capability/substrate (not just style), the facts-carry/texture-leaks split is **not medium-general** and the identity finding fails to replicate in code. Report each arm's pattern separately, then state whether they **agree** — a shared pattern across two different constructs/populations is the strength (§3.6 intro: conceptual replication); a divergence is a finding, not something to hide.
- **Code response variables:** (1) **correctness** — compiles + passes a small test; (2) **style-fidelity** — matches our conventions (public immutable vals & pure defs kept public, braces on long scopes, naming, latest-stable-Scala idioms — per our code-style memories + the Odersky/Regnell/Kerr style note); (3) **code-smells** — count via linter/heuristics; (4) **self-consistency** — does it code like our prior code.
- **Factors reused (no new design):** capability = CO4 / Fable + the ollama **code-specialist ladder** (`qwen2.5-coder` 0.5/1.5/3/7b, `deepseek-coder`, `starcoder2`, `codellama`, `granite-code`, `codegemma`) — the code axis set aside earlier now becomes central; substrate = full code-style substrate / empty / **scrambled**.
- **Negative control for style:** style-ablation (empty/scrambled code-style substrate) should collapse output toward the model's generic default → proves the style-carry is driven by *our substrate*, not the model's priors.
- **Tasks (two-tier coding-task battery):** **(i) micro-tasks** — a fixed set of held-constant small Scala tasks (pen-and-paper / muntabot-quest sized, C1-C5 in `047-instrument.md`) that *every* model incl. the 0.5b ones can attempt → the **capability-ladder floor**; **(ii) a PRD battery** — a set of small `PRD.md` specs (reusing our PRD-in-reqT work) as realistic **spec-to-code jobs** for the capable readers (CO4 / Fable / mid code-models) → **ecological validity**. A PRD's **acceptance criteria double as the correctness test** (no separate test authoring). **The two tiers carry DIFFERENT factors — they do not both hold both** (critique #7): the **micro-tasks (C1-C5) carry the capability ladder** (every model incl. 0.5b can attempt them → a shared capability axis across the whole ladder), while the **PRD battery carries substrate-completeness + ecological validity at the capable tier only** (a full-stack PRD one-shot by a 0.5b would floor, so PRDs route to capable readers). The capability axis is thus shared across the two arms only at the **micro-task grain**; the PRD arm is a capable-tier ecological probe, not a rung of the ladder.
- **Scoring is mostly objective** (compile / test / lint) → **higher conclusion validity** than the subjective identity probes; a short style/idiom rubric (2nd rater or auto-scorer) covers the taste dimension.
- **Builds on existing genscalator work:** the indent-vs-braces edit-cost experiment, the Odersky/Regnell/Kerr style note, blog 002/003.

#### PRD-battery spec — the goal-design scale (BR's RE course; Lauesen)
Each PRD is written in **reqT-lang** and structures its requirements along Lauesen's **goal-design scale** — four abstraction levels (source: BR's RE course L1, Fig 1.6A `The goal-design scale`; Sören Lauesen, *Software Requirements*, Addison-Wesley 2002 — a BR self-reference via his own course). Each PRD **re-engineers ≥2 requirements at each of the four levels**:

| Level | BR's definition (sv, verbatim) | Gloss (en) | Lauesen example |
|---|---|---|---|
| **Goal / Mål-nivå** | bakomliggande syfte, affärsmål, användarnytta, effekt, vinst | underlying purpose, business goal, user benefit, effect, profit | "Our pre-calculations shall hit within 5%" |
| **Domain / Domän-nivå** | sammanhang, omgivning, hur användarna och produkten **samverkar** för att ge nytta | context, environment, how users and the product **interact** to give benefit | "Product shall support cost recording and quotation with experience data" |
| **Product / Produkt-nivå** | externt observerbara funktioner och egenskaper | externally observable functions and properties | "Product shall have recording and retrieval functions for experience data" |
| **Design / Design-nivå** | specifik utformning av produktens innehåll | specific design of the product's content | "System shall have screen pictures as shown in app. xx" |

**n = 4** PRDs (maximum-variation case selection, Flyvbjerg — diverse *kinds* of coding job, NOT representative sampling), generated by the agent from BR's one-liners:
- **PRD1 — todo CRUD web app** (Scala 3.8 + Jetty; ScalaJS + Laminar client; no OAuth, server-generated password, mocked email). *Has goal/domain.*
- **PRD2 — terminal Sudoku** (load, play, backtracking solver). *Has goal/domain.*
- **PRD3 — CLI word-frequency tool** (read a text file, print top-N most frequent words). *Product/design only.*
- **PRD4 — arithmetic expression evaluator library** (parse + evaluate `2 + 3 * 4` with precedence). *Product/design only.*

**Rules across all four:** (a) **product + design levels mandatory**; goal/domain **optional** — present only where the app has a real purpose/context layer, which *itself* becomes a variation axis (*intent-carrying* PRDs vs *transcribe-the-spec* PRDs). (b) **Scala 3.8 fixed as a design requirement in every PRD** — holds language constant so the study varies task-kind, not language (clean control; note the residual per-model Scala-familiarity caveat). (c) difficulty spans the ladder (PRD3 floor → PRD1/PRD2 mid → PRD4 hard). (d) **each PRD includes a design-level requirement specifying HOW tests are done** — the test approach is part of the spec, so the model implements the tests and the PRD **self-scores correctness** objectively. (e) **NO quality (non-functional) requirements** — performance, security, scalability, usability are deliberately **out of scope** to keep the coding job tractable for dumb models; named as a scope bound / external-validity limit (the study measures **functional + style carry, not quality-requirement carry**).

**DIRECTIVE — design the PRD requirements for maximum INFORMATION YIELD (w.r.t. the RQs).** Do not write generic requirements. Apply information-oriented selection at the *requirement* grain, not just the app grain: craft each requirement to best **discriminate the study's DVs** — prefer requirements whose implementation most sharply reveals **style/idiom-carry vs correctness-carry** (RQ4), **intent-carry from goal/domain down into code** (RQ2/RQ4), and where a warped / dumb / substrate-starved reader is most likely to **diverge** from our conventions. A requirement every model implements identically regardless of substrate yields little; a requirement where our-style vs generic-default visibly *forks* yields much. **Hypothesize the yield before including a requirement.**

**Location of details:** the full **PRD1-4 specs** (the reqT models) + **per-PRD rationale** (why this app, which variation axis it covers, the hypothesized information yield, and the goal/domain-optional decision) live in `047-instrument.md` (Part 2, alongside the C1-C5 micro-tasks and the Q-battery). The plan carries the *design*; the instrument carries the *content*.

So building the PRD instruments is **human-gated** — like the joint surf and the feasibility gate — the agent proposes the specs, BR ratifies them.

**PRD naturalism (methodology).** The PRDs are CO4-generated, incomplete, **structurally reviewed by BR but only lightly content-reviewed** — deliberately **realistic**: real coding starts from imperfect, incomplete requirement input under pressure (a stressed dev org, premature coding from partial specs). An **ecological-validity strength we own**, not a flaw to hide (it also pushes the model *beyond pure transcription* — it must infer unstated details, and whether it infers them in *our* style is part of the signal, which partly answers the "full condition = instruction-following not reconstruction" critique). **Constraint:** we do NOT deliberately inject confusing, contradictory or "bad" requirements — the PRDs are **honest-but-incomplete, not adversarially sabotaged**. The "degraded input" manipulation lives at the **substrate** level (full / empty / scrambled = Factor B), NOT in planted-bad PRD requirements — keeping bad-reqs out of the PRD isolates Factor B cleanly.

**Scoring / assessment — CO4 as expert reviewer (methodology; open research task).** Beyond the mechanical checkpoints, a smart model (**CO4**) reviews each dumb-model program to assess **dev-relevant qualities** in the **before/after-warp** comparison (loaded / full-substrate = "before"; ablated / dumb reader = "after warp") — mirroring a senior dev reviewing junior-or-model code (realistic). **Which qualities is an OPEN research task delegated to the smart model to develop during the research** — candidates: idiomaticity / our-style adherence, readability, maintainability, appropriate abstraction, smell-freedom, "would a dev want to maintain this". **Anti-circularity guardrails (from the adversarial critique, #2):** CO4 co-authored our style AND is a top subject, so the review MUST be **blind-to-condition** (strip which model / which substrate produced the code); mechanically-parseable checkpoints (uses `enum`, uses `Option`, `val` not `var`, braces) are scored by a **linter / AST pass, not the LLM judge** — reserve CO4-judgment for the residual *subjective* qualities, with a **self-preference control** (does CO4 rate CO4-authored code above a blind-human baseline? report the delta).

The fresh reviewer corrected author-blindness **both ways** (found the PRD1/PRD2 misses AND overruled the author's PRD4 over-flag) — corroborating the independent-reviewer method.

**Why the four levels matter for coding-fidelity:** a coding job lives mostly at **product/design** level (what to build, how specifically); **goal/domain** are the *intent and context the implementation must honor*. A PRD spanning all four tests whether a warped/dumb reader carries **high-level intent down into our-style code**, not merely transcribes a design spec (Lauesen's own point: the level you specify at depends on the "supplier" — here, the model under test). **reqT-lang** makes requirements parseable, so acceptance criteria can be machine-checked (the PRD self-scores the coding job).

| Goal | `Goal` |
| Domain | `Feature` |
| Product | `Function` |
| Design | `Design` |

Example (reqT-lang, from the slide):
```
Goal: accuracy has Spec: "Our pre-calculations shall hit within 5%"
```

So each PRD is a reqT model: **≥2 `Goal` + ≥2 `Feature` + ≥2 `Function` + ≥2 `Design`** entities, each with a `Spec`. The product/design `Spec`s become the coding job's checkable acceptance criteria.
