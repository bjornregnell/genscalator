# Todo CRUD web app — Product Requirements (reqT-lang)

## Goal level

* Goal: captureTasks has
  * Spec: Users reliably capture and track their personal tasks so nothing is forgotten.
  * Why: the user benefit is fewer dropped tasks.
* Goal: lowFrictionAccess has
  * Spec: A user reaches their task list from any browser with minimal onboarding and no third-party account.

## Domain level

* Feature: manageOwnTaskList has
  * Spec: A signed-in user creates, views, edits, completes and deletes their own tasks via a web browser.
* Feature: passwordByEmail has
  * Spec: A user obtains access by requesting a one-time password delivered to their email; no OAuth.
* Feature: manageOwnTaskList helps Goal: captureTasks
* Feature: passwordByEmail helps Goal: lowFrictionAccess

## Product level

* Function: todoCrud has
  * Spec: The product exposes create, read, update, delete and toggle-complete on todo items; each item has a title, a done flag and a creation timestamp.
* Function: serverGeneratedPassword has
  * Spec: On an access request the product generates a random password server-side and does not reveal it in the HTTP response; it is delivered only via the email channel.
* Function: ownDataOnly has
  * Spec: A user can read and modify only their own todos.
* Function: todoCrud helps Feature: manageOwnTaskList
* Function: serverGeneratedPassword helps Feature: passwordByEmail
* Function: ownDataOnly helps Feature: manageOwnTaskList

## Design level

* Design: scalaStack has
  * Spec: Server-side Scala 3.8 on Jetty; client-side Scala.js with Laminar; the shared model is cross-compiled; no OAuth.
* Design: immutableDomain has
  * Spec: Todo and User are immutable Scala 3 case classes with public vals; a state change produces a new value; no mutable fields.
* Design: functionalErrors has
  * Spec: Service operations return Option or Either for not-found, unauthorized and validation outcomes; no exceptions for control flow and no null.
* Design: mockedEmailSeam has
  * Spec: Email delivery is an injected trait with a mock that records the last password sent per address, so a test asserts delivery without SMTP.
* Design: testApproach has
  * Spec: mUnit tests cover the server-side domain, CRUD service, password generation and the mocked-email seam (create, read, update, delete, toggle-complete, own-data-only, password generated and delivered). The Laminar client is not unit-tested.
* Design: scalaStack helps Function: todoCrud
* Design: immutableDomain helps Function: todoCrud
* Design: functionalErrors helps Function: todoCrud
* Design: mockedEmailSeam helps Function: serverGeneratedPassword
* Design: testApproach verifies Function: todoCrud
* Design: testApproach verifies Function: serverGeneratedPassword
